## 3. 핵심 아키텍처

### 3.1 청크 시스템 (Chunk System)

월드를 일정한 크기의 청크 단위로 분할하여 관리.

```
월드 좌표계 (World Coordinates)
  - 플레이어 위치는 월드 전체 좌표 (x, y, z) 로 표현

청크 (Chunk)
  - 크기: 16 × 16 × 64 (가로 × 세로 × 높이)
  - 하나의 청크는 Uint8Array(16 * 16 * 64 = 16,384개 블록)
  - 청크 좌표: (chunkX, chunkZ) = (floor(worldX / 16), floor(worldZ / 16))

블록 데이터 접근
  - 로컬 인덱스: index = x + z * 16 + y * 256
  - 청크 내 블록: chunk.blocks[index] = blockId

월드 블록 접근 API (★ 핵심 - 이웃 청크 참조용)
  - World.getBlock(worldX, worldY, worldZ): worldX/Z로 청크를 찾아 로컬 인덱스로 변환
  - 로드되지 않은 청크 / 월드 경계 밖 → AIR(0) 반환 (안전장치)
  - World.setBlock(worldX, worldY, worldZ, blockId): 블록 변경 + 해당 청크 dirty 마킹

청크 컬렉션 관리
  - 자료구조: Map<string, Chunk> (키: `"chunkX,chunkZ"`)
  - 키 생성: `${Math.floor(x/16)},${Math.floor(z/16)}`
```

**청크 매니저 동작:**
1. 플레이어 중심 반경 N칸 내의 청크를 항상 로드
2. 벗어난 청크는 언로드 (메시 제거, 데이터는 캐시 또는 폐기)
3. 프레임당 1~2개 청크만 생성하여 프레임 드랍 방지
4. **스폰 시**: 플레이어 주변 청크를 동기적으로 선로드 (낙하 방지)

### 3.2 지형 생성 (Terrain Generation)

Simplex Noise를 여러 옥타브로 중첩하여 자연스러운 지형 생성.

```
높이 계산:
  height = BASE_HEIGHT
         + noise(x * 0.01, z * 0.01) * AMPLITUDE        // 메인 지형
         + noise(x * 0.05, z * 0.05) * AMPLITUDE * 0.3   // 디테일
         + noise(x * 0.1,  z * 0.1)  * AMPLITUDE * 0.1   // 미세 요철

블록 배치 (높이별):
  y == height        → Grass (잔디)
  y < height && y > height - 3 → Dirt (흙)
  y <= height - 3    → Stone (돌)
  y == 0             → Bedrock (기반암, 부수기 불가)
```

**확장 가능성 (Phase 2):**
- 나무 생성 (트리 구조 패턴)
- 물 (해수면 아래)
- 동굴 (3D 노이즈)
- 광물 (석탄, 철, 다이아몬드 - 깊이별 확률 분포)

### 3.3 메시 생성 (Mesh Generation) ★ 핵심 최적화

**핵심 원칙:** 모든 블록을 개별 큐브 메시로 렌더링하면 끔찍하게 느리다.  
→ 청크 단위로 하나의 병합된 메시(BufferGeometry)를 생성한다.

**노출 면 컬링 (Face Culling):**
```
각 블록의 6면(±X, ±Y, ±Z)에 대해:
  인접한 블록이 비어있거나(air) 투명하면 → 그 면을 메시에 추가
  인접한 블록이 불투명 고체면 → 그 면을 스킵 (어차피 안 보임)
```

**결과:** 지하 블록은 전혀 렌더링되지 않음. 표면만 렌더링 → 극적인 성능 향상.

**부분 메시 재생성 (★ 단일 블록 변경 최적화):**
```
블록 변경 시 (setBlock):
  1. 해당 청크를 dirty 마킹
  2. 변경된 블록이 청크 경계(0 또는 15)에 있으면 이웃 청크도 dirty 마킹
  3. 다음 프레임에서 dirty 청크만 메시 재생성
  → 전체 청크 재생성이 아닌 dirty 큐 기반 배치 처리
```

**투명 블록 분리 렌더망:**
```
- 불투명 블록 (흙, 돌, 나무 등): 단일 메시, depthWrite=true
- 투명 블록 (물, 나뭇잎): 별도 메시, transparent=true, depthWrite=false
- 청크당 최대 2개 메시 (불투명 + 투명)
- 투명 메시는 별도 Material로 렌더링 순서 제어
```

**Phase 2 심화 (Greedy Meshing):**
- 같은 타입의 인접한 면을 하나의 큰 사각형으로 병합
- 폴리곤 수를 더욱 대폭 줄임
- 구현 복잡도가 높으므로 MVP 이후 적용

### 3.4 레이캐스트 (Voxel Raycasting)

플레이어가 바라보는 블록을 식별하기 위한 DDA 알고리즘.

```
입력: 카메라 위치 + 시선 방향 벡터
출력: 타격한 블록의 월드 좌표 + 어느 면을 hit 했는지 (설치용)

방식: DDA (Digital Differential Analyzer)
  - 시선을 따라 복셀 경계를 하나씩 순회
  - 최대 거리 (예: 5블록) 내에서 첫 번째 고체 블록 반환
  - hit한 면의 법선 벡터로 "설치할 위치" 계산
```

## 4. 게임 디자인

### 4.1 블록 타입

| ID | 이름 | 색상/텍스처 | 투명 | 특이사항 |
|----|------|------------|------|----------|
| 0 | Air | - | ✅ | 빈 공간 |
| 1 | Grass | 상단 초록, 측면 흙+잔디 | ❌ | 지형 표면 |
| 2 | Dirt | 갈색 | ❌ | 지표층 |
| 3 | Stone | 회색 | ❌ | 지하층 |
| 4 | Wood | 나뭇결 패턴 | ❌ | 건축용 |
| 5 | Leaves | 짙은 초록 | 반투명 | 나뭇잎 |
| 6 | Sand | 밝은 노랑 | ❌ | 해변 |
| 7 | Water | 파랑 (애니메이션) | 반투명 | 해수면 |
| 8 | Bedrock | 검정 회색 | ❌ | 부숴지지 않음 |

### 4.2 조작키

| 조작 | 키 | 설명 |
|------|------|------|
| 이동 | `W` `A` `S` `D` | 전/좌/후/우 |
| 점프 | `Space` | 1블록 높이 점프 |
| 시점 | 마우스 이동 | Pointer Lock 모드 |
| 블록 파괴 | 좌클릭 | 바라보는 블록 제거 |
| 블록 설치 | 우클릭 | 바라보는 면 옆에 설치 |
| 블록 선택 | `1`~`8` / 마우스 휠 | 핫바 슬롯 변경 |
| 고속 이동 | `Shift` | 크리에이티브 모드 하강 |
| 고속 상승 | `Space` (더블탭) | 크리에이티브 모드 비행 |

### 4.3 물리 시스템

```
중력:     -30 m/s² (게임 감각에 맞춘 가속도)
이동 속도: 5 m/s (걷기)
점프 높이: ~1.2블록 (1블록 위로 올라갈 수 있게)
터미널 속도: -50 m/s (낙하 속도 제한)

충돌 감지 (AABB - Axis Aligned Bounding Box):
  - 플레이어를 0.6×1.8×0.6 크기의 박스로 취급
  - 이동 후 각 축별로 블록 충돌 검사 후 위치 보정
  - 축별 분리 검사 (X → Z → Y 순)로 슬라이딩 자연스럽게

안전장치 (★ 검토 반영):
  - 미로드 청크에서 충돌 검사 시: 월드 API가 AIR 반환 → 플레이어 낙하
    → 낙하 한계 Y < -10 도달 시 가장 가까운 로드된 청크 표면으로 텔레포트
  - 스폰 시: 주변 3×3 청크 동기 생성 후 스폰 (비동기 로딩 전 완료)
  - 고속 이동(비행) 시: 이동 거리가 1블록 초과면 서브스텝으로 분할 검사 (터널링 방지)
```

## 7. 성능 고려사항

## 10. 검토 반영사항 (Momus Review Conditions)

> Momus 에이전트 검토 결과(GO with conditions) 반영 — 구현 시작 전 5가지 최소 조건

### 조건 1: 경계 케이스 대응 ❌→✅

| 케이스 | 대응 |
|--------|------|
| 미로드 청크 낙하 | 스폰 시 동기 로드 + 낙하 한계 도달 시 텔레포트 |
| 자기 위치 블록 설치 | 설치 위치와 플레이어 AABB 교차 검사 → 거부 |
| Y=0 이하 설치 | 설치 금지 + 기반암 파괴 불가 |
| 월드 경계 | X/Z 좌표 제한 (±10000), 경계 외 블록 접근 시 AIR 반환 |
| 고속 이동 터널링 | 이동 거리 > 1블록 시 서브스텝 분할 검사 |

### 조건 2: 청크 관리 자료구조 명시

```typescript
// World 클래스 내부
private chunks: Map<string, Chunk>  // 키: "chunkX,chunkZ"

function chunkKey(cx: number, cz: number): string {
  return `${cx},${cz}`;
}
```

### 조건 3: 메시 갱신 최적화 (부분 업데이트)

```
블록 변경 → dirty 큐에 청크 등록
경계 블록 변경 → 이웃 청크도 dirty 등록
매 프레임: dirty 큐에서 N개 청크만 메시 재생성 (프레임 드랍 방지)
```

### 조건 4: 청크 경계면 처리 (World.getBlock API)

```typescript
// 월드 좌표 → 블록 ID (이웃 청크 참조 포함)
World.getBlock(worldX, worldY, worldZ): number
// 월드 좌표 → 블록 설정 + dirty 마킹
World.setBlock(worldX, worldY, worldZ, blockId): void
```

### 조건 5: 투명 블록 렌더링 분리

- 불투명 메시: `depthWrite=true, transparent=false`
- 투명 메시: `depthWrite=false, transparent=true`
- 청크당 최대 2개 메시 (불투명 + 투명)
- Material 분리로 렌더링 순서 보장
